# CLAUDE.md

Guidance for agents working in OpenUSDZEditor — a native macOS USDZ viewer/editor (SwiftUI + RealityKit + embedded Python/`usd-core`).

## Concurrency (read first)

Multiple agents work this repo at the same time. **Never run `git stash`** — it mutates shared working-tree state and clobbers other agents' uncommitted changes. Isolate your work on a topic branch (or a worktree) instead. Never commit to `main` directly; land changes via PR with green CI.

## Testing — 100% coverage

New behavior ships with tests, and logic modules hold a **100% line-coverage floor** (95% for a few). Coverage is CI-enforced per module via `scripts/coverage-gate.sh`; do not lower a floor or add override labels. MeshKit additionally runs a fuzz corpus (`FuzzCorpus.swift`) — extend it for fuzz-sensitive changes. See `specs/testing.md` for per-module policy.

Run the full local check set (mirrors CI) before opening a PR:

```sh
bash scripts/dependency-lint.sh       # one-directional dependency rules
bash scripts/module-governance.sh     # module governance gate
bash scripts/fetch-python-runtime.sh  # one-time: usd-core runtime
bash scripts/test-all.sh --coverage   # all package tests
bash scripts/coverage-gate.sh         # coverage floors + fuzz corpus
bash scripts/roundtrip-gate.sh        # round-trip invariants over the bridge mini-corpus
```

## Architecture — modern Swift (2026)

Swift 6.0, strict concurrency, macOS 14+. Follow current best practices: value types and `actor`/`Sendable` isolation over shared mutable state, `async/await` (no callback pyramids or legacy GCD), dependency injection over singletons, protocol-oriented extension points, and `@Observable`/`@MainActor` SwiftUI. Keep new code `Sendable`-clean rather than silencing data-race diagnostics.

The workspace is modular SPM packages under `Packages/` with **strictly one-directional dependencies** enforced by `scripts/dependency-lint.sh`. Layering (a module may only import what its policy allows):

- `USDCore`, `MeshKit`, `DicyaninDesignSystem` — leaf modules, zero internal deps. `USDCore`/`MeshKit` stay pure Swift (no UI/GPU/Python).
- `USDBridge`, `ConversionKit`, `ValidationKit`, `ScriptingKit` → `USDCore`.
- `ViewportKit` → USDCore, MeshKit. `EditingKit` → USDCore, ValidationKit, MeshKit.
- `AgentMCP` → the kits (never `EditorUI`). `EditorUI` → all kits + DesignSystem. `App`/`CLI` → top of the graph.

Adding a package requires a matching policy entry in `dependency-lint.sh` **and** an update to `specs/architecture.md` — an ungoverned module is itself a lint failure. Specs in `specs/` are the contract; keep them in sync with code.

## SwiftUI & UI code (EditorUI, DesignSystem)

Write UI as an expert SwiftUI 6 + macOS architect would. These rules are mandatory for any SwiftUI/UI code:

- **Latest SwiftUI patterns only.** Use `NavigationStack`/`NavigationSplitView` (never `NavigationView`), the `@Observable` macro for models (never `ObservableObject`/`@Published`), `@Environment` for dependency access, `@State` for view-owned state, and `@Bindable` where two-way bindings to `@Observable` types are needed.
- **No deprecated APIs.** Do not use `NavigationView`, `ObservableObject`, `@StateObject`/`@EnvironmentObject`, `onChange(of:perform:)` (use the two-parameter closure form), or any API marked deprecated on macOS 14+. If a modern replacement exists, use it.
- **Clean architecture — separate concerns.** Keep Views, ViewModels, Models, and Services in distinct types/files:
  - *Views* stay declarative and thin — layout and binding only, no business logic or I/O.
  - *ViewModels* are `@Observable @MainActor` types holding view state and orchestration; they depend on Services via injection, not on concrete singletons.
  - *Models* are plain `Sendable` value types.
  - *Services* encapsulate side effects (bridge/Python calls, persistence, networking) behind protocols so they can be injected and tested.
- **Dependency injection over singletons**, consistent with the repo-wide rule. Inject Services into ViewModels; expose shared dependencies through `@Environment` rather than global accessors.
- **Concurrency.** UI state mutation is `@MainActor`; use `async/await` and `.task {}` for lifecycle-bound async work (no GCD, no completion-handler pyramids). Keep types `Sendable`-clean.
- **Production-ready, well-commented code.** Comment the *why* (non-obvious layout, concurrency, or bridge constraints), not the obvious. No force-unwraps in view code; handle loading/empty/error states explicitly.

## Layout

- `App/` — macOS app shell. The `.xcodeproj` is generated by XcodeGen from `project.yml` (git-ignored); edit `project.yml`, then `bash scripts/generate-xcodeproj.sh`. SPM packages remain the source of truth for library code.
- `CLI/` — the `openusdz` command-line tool (same pipeline as the app; also serves the MCP server).
- `Packages/` — Swift library packages.
- `Resources/Python/` — bundled `usd-core` bridge scripts.
- `specs/`, `PRD.md`, `ROADMAP.md` — design contract and phase status.

## PRs

**Every PR targets `main`. No exceptions.** Branch a topic branch off `main`, PR against `main`, squash-merge into `main`. Never open, base, or merge a PR against another feature/topic branch — not even to "keep the diff scoped" or because the base isn't on `main` yet. If the work depends on an unmerged branch, land that branch to `main` first (or rebase your work onto `main`), then open your PR against `main`. If you ever find yourself about to set a PR `--base` to anything other than `main`, stop and ask the user.

Use a conventional-commit-style title (e.g. `feat(viewport): …`, `fix(meshkit): …`). Both `lint` and `test` CI jobs must pass before merge.
